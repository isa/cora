# Phase 3.3: Component Preview Canvas - Research

**Researched:** 2026-05-22
**Status:** Ready for planning

## Research Question

What does the planner need to know to implement `cora preview` as a
workbench-first browser SPA without polluting the pure renderer component
surface or creating throwaway dev-server infrastructure before Phase 4?

## Key Findings

### Dev server strategy

- Use Vite's JavaScript API for the preview command rather than shelling out to
  `vite`. The official API exposes `createServer(inlineConfig)` returning a
  `ViteDevServer` with `listen()`, `printUrls()`, `resolvedUrls`, and
  `bindCLIShortcuts()`. This maps cleanly to `cora preview` and Ctrl-C cleanup.
- Configure Vite with `configFile: false` and an explicit preview root so Cora
  does not depend on a user project's `vite.config.*`.
- Vite's `preview()` API serves a production build and is useful for packaged
  asset smoke tests, but the interactive development command should use
  `createServer()` so source edits can hot-reload during Cora development.
- Vite supports `preview.open`, `preview.port`, and auto-selecting the next
  available port when the configured port is already used. The Cora command
  should print the actual resolved URL rather than assuming a fixed port.

### Build and package strategy

- The npm tarball must include the built preview SPA. `packages/cora/package.json`
  currently ships `dist` and `.npmrc`, and the build script copies renderer
  assets into `dist/renderer/`. Phase 3.3 must extend this so preview HTML/JS/CSS
  assets land under `dist/preview/` or another deterministic directory that the
  CLI can serve after `npm install cora`.
- Keep the CLI entry as `dist/cli.js`. Add a `preview` command module registered
  from `packages/cora/src/cli/index.ts`.
- Prefer a shared preview server helper that accepts a root/outDir and can be
  reused by Phase 4's future `cora serve`; do not hard-code a one-off Vite call
  directly inside the command action.
- Vite build assets should use a relative base (`base: './'`) for packaged
  static serving unless the planner proves a better package-local serving model.

### Preview source boundaries

- Preview UI code, manifest metadata, scenario definitions, and control schemas
  belong in `packages/cora/src/preview/` or a similarly preview-local source
  tree.
- Do not add UI metadata to `packages/cora/src/renderer/components/`. That layer
  remains pure React/SVG and is already the public renderer contract.
- The built-in pack should consume the renderer component barrel if technically
  feasible. Inside source tests this likely means importing from
  `../renderer/components/index.js`; package-level examples can verify the public
  `cora/renderer/components` subpath.

### Workbench interaction model

- Treat selected items as nodes only: exactly primary and secondary. Lines and
  groups are relationship/context objects.
- Drag state can be entirely client-side React state. It should update node
  coordinates, line endpoint calculations, and attachment overlay slots live,
  but must not write YAML or mutate disk.
- Full ELK layout is unnecessary for the preview MVP. Use deterministic scene
  coordinates for scenarios and recompute simple edge endpoints from current
  node boxes as nodes move.
- To support multiple connections on one side, model attachment as slots on a
  side/region, not a single top/right/bottom/left point. A helper can distribute
  `N` slots evenly across the side and label them `top-1`, `top-2`, etc.

### Control schema

- `PackManifest` should contain component ids, labels, families, default props,
  control schema, and supported scenario ids. Keep it internal to preview in
  this phase.
- Use discriminated control definitions so the UI can prevent invalid prop
  values: enum/select, boolean, color, bounded number, text, and size
  preset/explicit dimensions.
- Reset props and positions when switching component or scenario. This avoids a
  session persistence layer and keeps v1 predictable.
- Connection controls are separate from node controls and only appear for
  connected scenarios.

### Existing code to reuse

- `packages/cora/src/renderer/components/index.ts` exports the catalog surface:
  nodes, `Group`, `Line`, marker helpers, icon types, style helpers, and shared
  types.
- `packages/cora/src/renderer/components/types.ts` defines
  `BoxStyleProps`, `PageNodeProps`, `IssueNodeProps`, size presets, border
  styles, and other prop vocabulary the control schema should mirror.
- `packages/cora/src/renderer/components/lines/Line.tsx` is already a direct
  line primitive driven by explicit points and markers.
- `packages/cora/src/renderer/components/edges/edgePath.ts` contains useful
  marker-clearance and path rounding behavior for the current layouted-edge
  renderer. Preview-specific endpoints can stay simpler, but should preserve
  the same conceptual marker clearance.
- `packages/cora/src/web/index.ts` is currently a placeholder, so the preview
  can establish the web/preview module layout.
- `packages/cora/tests/renderer/components.test.tsx` already verifies renderer
  components through `renderToStaticMarkup`; Phase 3.3 should add preview tests
  for manifests, control coercion, slot distribution, and scenario state.

## Recommended Plan Shape

1. **Infrastructure + pack contract:** add `cora preview`, Vite server/build
   plumbing, preview source/build entries, and the internal `PackManifest` type.
2. **Workbench model + controls:** build built-in pack metadata, control schema,
   scenario state, two-node selection, connection panel, and reset behavior.
3. **Canvas + overlay:** render the workbench, draggable nodes, line updates,
   generic group wrapper, scenario presets, computed attachment slots, and
   responsive UI states.
4. **Packaging + verification sweep:** ensure built preview assets ship in the
   npm tarball, add smoke/unit tests, update docs/AGENTS, and verify build,
   tests, and clean-install behavior.

## Validation Architecture

Use Vitest for all automated checks and add focused preview tests before or
alongside implementation. The phase should be validated at three levels:

| Layer | What to prove | Suggested checks |
|-------|---------------|------------------|
| Unit | `PackManifest`, control schema, scenario defaults, reset behavior, slot distribution | `bun x vitest run tests/preview/*.test.ts*` |
| Integration | `cora preview` command registers, starts a server, prints a localhost URL, and closes cleanly | CLI test with a stubbed/open-disabled preview server helper |
| Packaging | Built preview assets are emitted and included in package tarball | `bun run build`, `npm pack --dry-run` or existing clean-install smoke extension |

Required automated commands:

- `bun run build`
- `bun x vitest run`
- `bun run cora preview --help` or equivalent CLI help assertion after build
- A package smoke check that proves `dist/preview/` assets are present in the
  packed package.

Manual/browser verification is still required before phase closeout:

1. Start `bun run cora preview`.
2. Confirm the opened localhost page shows catalog, canvas, and inspector.
3. Select primary and secondary nodes.
4. Drag both nodes and verify the connection line and attachment slots update.
5. Toggle overlay labels.
6. Switch scenarios and confirm controls reset to defaults.

## Open Risks

- Vite is present as an optional peer through Vitest/tsdown in the lockfile, but
  `cora preview` may require declaring `vite` as a direct dependency so the
  published package can import it reliably at runtime.
- Opening the default browser from a CLI may need opt-out flags for CI/tests;
  planners should include a non-opening path or internal test seam.
- The SDK currently mis-detects the decimal Phase 3.3 directory. Planning and
  execution should keep using `.planning/phases/3.3-component-preview-canvas`
  unless the GSD metadata is corrected separately.
