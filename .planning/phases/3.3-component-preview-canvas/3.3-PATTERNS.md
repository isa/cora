# Phase 3.3: Component Preview Canvas - Pattern Map

**Generated:** 2026-05-22
**Status:** Ready for planning

## Closest Existing Analogs

| New area | Closest analog | Pattern to reuse |
|----------|----------------|------------------|
| CLI command registration | `packages/cora/src/cli/index.ts`, `packages/cora/src/cli/commands/render.ts` | Add focused command modules and register from the single CLI root. Validate options with Commander before doing work. |
| Build entries | `packages/cora/tsdown.config.ts` | Add explicit build entries and deterministic output directories rather than relying on deep imports. |
| Package assets | `packages/cora/package.json` build script | Extend the existing copy step that moves renderer assets into `dist/`. |
| Component catalog imports | `packages/cora/src/renderer/components/index.ts` | Consume the public component barrel and keep preview metadata outside renderer components. |
| Component prop vocabulary | `packages/cora/src/renderer/components/types.ts`, `packages/cora/src/renderer/components/styles.ts` | Mirror `BoxStyleProps`, `ComponentSize`, `BorderStyle`, `PageNodeType`, `IssueIconType`, and size presets in control metadata. |
| Line/marker controls | `packages/cora/src/renderer/components/lines/Line.tsx`, `packages/cora/src/renderer/components/lines/markers.tsx` | Use explicit points, `LineStyle`, `MarkerType`, and centralized marker definitions. |
| Edge geometry | `packages/cora/src/core/edgeGeometry.ts`, `packages/cora/src/renderer/components/edges/edgePath.ts` | Reuse point/segment concepts and marker clearance behavior; preview endpoint computation may remain simpler and deterministic. |
| Tests | `packages/cora/tests/renderer/components.test.tsx`, `packages/cora/tests/pdf/*.test.ts` | Use Vitest, pure helper assertions, and `renderToStaticMarkup` where useful. Avoid browser-only test dependencies unless deliberately added. |
| Clean install smoke | `packages/cora/tests/smoke/clean-install.sh` | Extend or add a package smoke check for preview assets without requiring browser interaction. |

## File Role Guidance

### Preview source

- `packages/cora/src/preview/` owns preview UI, pack manifest, control schema,
  scenario state, drag state, and attachment-slot helpers.
- Do not put preview metadata in `packages/cora/src/renderer/components/`.
- `packages/cora/src/web/index.ts` may re-export or route to preview/web code if
  the planner chooses that as the web module surface.

### CLI and server

- `packages/cora/src/cli/commands/preview.ts` should mirror the command-module
  style used by `render.ts`, but should avoid diagram YAML parsing.
- A reusable server helper should live under preview/web source and expose a
  test seam for `open: false` / no browser launch.
- The helper should print or return the actual Vite resolved URL.

### Tests

- `packages/cora/tests/preview/*.test.ts*` should cover pure preview logic:
  manifest completeness, control defaults, invalid value prevention, scenario
  reset behavior, attachment slot distribution, and line point recomputation.
- CLI tests should prove command registration/help and server helper invocation
  without starting a long-running process in CI.

## Implementation Constraints

- No YAML persistence.
- No external pack loading.
- No provider/service icon resolution.
- No UI registry dependency.
- No viewport-scaled type or decorative landing-page treatment.
- Phase 4 should be able to reuse the dev-server abstraction.
