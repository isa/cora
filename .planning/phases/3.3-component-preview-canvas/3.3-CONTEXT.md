# Phase 3.3: Component Preview Canvas - Context

**Gathered:** 2026-05-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Build `cora preview` as a browser-based component workbench for the renderer
component catalog. The user can pick built-in node components, compare up to
two selected nodes, drag them on a canvas, edit their typed props, inspect how
connections attach between them, and view simple scenario contexts such as
isolated, connected, grouped, and grouped + connected.

This phase owns the preview SPA, local dev-server command, built-in preview
pack manifest, live controls, drag-only preview positioning, scenario presets,
and designer-facing line attachment overlay.

This phase does not own YAML persistence, full diagram editing, file watching,
saved canvas state, external extension loading, or provider/service icon
resolution. Those remain Phase 4 and Phase 5 work.

</domain>

<decisions>
## Implementation Decisions

### Preview surface and navigation
- **D-01:** The preview is workbench-first. The canvas is the center of the
  experience, with sidebar and controls supporting small compositions.
- **D-02:** "Selected components" means nodes only. Lines/connections and
  groups are inspectable context, but they do not count as selected components.
- **D-03:** The workbench supports exactly two selected nodes: a primary node
  and a secondary node.
- **D-04:** Both selected nodes can be dragged on the canvas and have editable
  attributes.
- **D-05:** Dragging is preview-only and ephemeral. It exists to inspect
  positions, connection behavior, and attachment changes, not to persist layout.
- **D-06:** Dragging selected nodes should recompute renderer-like line
  attachments against node anchors as positions change.
- **D-07:** The user should be able to select a node and see it connected to a
  secondary node, connected to different catalog node types, and shown inside a
  group context.

### Control model
- **D-08:** Controls are generated from each selected node component's typed
  props, not only from a curated designer subset.
- **D-09:** Specialized component props must be exposed where applicable,
  including examples such as `PageNode.type`, `IssueNode.icon`, icon colors,
  skeleton colors, size, border style, and other catalog-specific props.
- **D-10:** Primary and secondary selected nodes use side-by-side prop panels.
- **D-11:** Lines/connections use a separate connection panel. The connection
  panel controls relationship context such as line style, stroke color, stroke
  width, markers, and connection mode.
- **D-12:** Controls prevent invalid values through constrained UI inputs:
  selects for enums, sliders or bounded number inputs, color pickers for colors,
  text fields where free text is valid, and structured size controls for preset
  vs explicit dimensions.
- **D-13:** Switching components or scenes resets controls to defaults.

### Combination presets
- **D-14:** Presets are scenario-based around the selected primary node rather
  than a separate static gallery.
- **D-15:** Required scenarios are: isolated primary node, primary connected to
  secondary node, primary inside a group, and primary inside a group connected
  to secondary.
- **D-16:** Connected scenarios include marker and line variants.
- **D-17:** The preview also includes component-family showcases for page, app,
  issue, and icon-style nodes.
- **D-18:** Connected scenarios use an explicit secondary node picker from the
  component catalog.
- **D-19:** Marker and line presets set starting values, then the separate
  connection panel can edit line style, stroke color, stroke width, markers,
  and connection mode.
- **D-20:** Group scenarios use one generic group wrapper.

### Pack and extension boundary
- **D-21:** Phase 3.3 defines a real built-in `PackManifest` and represents
  built-in renderer components as a pack.
- **D-22:** Phase 3.3 does not load external packs or implement extension
  installation.
- **D-23:** `PackManifest` describes components plus scenarios: component ids,
  labels, family/category, default props, control schema, and supported scenario
  presets per component.
- **D-24:** `PackManifest` must not include extension install/version/provider
  metadata yet.
- **D-25:** Preview-specific manifest and control metadata lives in
  preview-local source such as `packages/cora/src/preview/`, not in
  `packages/cora/src/renderer/components/`.
- **D-26:** `PackManifest` is an internal preview contract in Phase 3.3. It
  should be typed and structured enough for Phase 5 to build on, but it should
  not be exported as a public package subpath yet.
- **D-27:** The built-in pack should consume the public renderer component
  barrel if technically feasible, so preview exercises the same component
  surface future packs will use.

### Line attachment overlay
- **D-28:** The overlay shows computed attachment slots for the current scene,
  not only abstract anchor sides.
- **D-29:** The overlay must not imply one connection per side. If multiple
  lines attach on the same side, slots are distributed and labeled such as
  `top-1`, `top-2`, and `top-3`.
- **D-30:** The overlay also shows subtle side guides for top/right/bottom/left
  regions, so users can understand both the side region and the actual computed
  slots.
- **D-31:** Slot labels are toggleable. Dots and subtle side guides can remain
  visible while labels are turned on or off.
- **D-32:** The overlay is designer-facing only in Phase 3.3. It should not
  include coordinates, path data, routing metadata, or a renderer-debug mode.
- **D-33:** Slots and labels update live while selected nodes are dragged.

### the agent's Discretion
- Exact visual layout, spacing, colors, and responsive breakpoints are open,
  constrained by the workbench-first model and the project's existing renderer
  aesthetic.
- Exact control component choices are open as long as invalid prop values are
  prevented.
- Exact default props, preset names, line variants, and generic group wrapper
  styling are planner/executor decisions.
- Exact dev-server implementation is open, but it should seed reusable
  infrastructure for Phase 4 rather than being a dead-end one-off.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/cora/src/renderer/components/` already contains the public catalog
  components, shared style helpers, line primitive, markers, and icon slot
  types that the preview should use.
- `packages/cora/src/renderer/components/index.ts` is the supported barrel for
  renderer components and should be the preview's preferred component surface
  if technically feasible.
- `packages/cora/src/renderer/components/edges/edgePath.ts` already computes
  line path data and marker points for layouted edges; it is a useful reference
  for preview attachment behavior.
- `packages/cora/src/web/index.ts` exists only as a Phase 4 placeholder, so
  Phase 3.3 can establish the preview/web source structure.
- `packages/cora/package.json` already has a CLI bin and package export map;
  preview assets must be included in packaged output if `cora preview` is
  expected to work after install.

### Established Patterns
- The renderer component layer stays pure React/SVG and DOM-independent.
  Preview UI metadata should not be added to `renderer/components/`.
- Package consumers use `cora/renderer/components`; top-level `cora` stays free
  of component exports.
- CLI commands are registered from `packages/cora/src/cli/index.ts` through
  focused command modules under `packages/cora/src/cli/commands/`.
- The project uses Bun, tsdown, React 19, Vitest, and existing render-golden
  tests.

### Integration Points
- Add a `cora preview` command in the CLI without assuming deferred commands
  like `cora serve`, `cora ext`, or `cora doctor` exist.
- Add preview-local source such as `packages/cora/src/preview/` for the
  workbench UI, built-in pack manifest, control schema, scenario presets, and
  preview-only state.
- Dev-server and asset packaging choices should be designed so Phase 4 can
  reuse them for `cora serve`.
- The built-in pack should be internal in Phase 3.3 but shaped so Phase 5 can
  later add extension-loaded packs without rewriting the preview model.

</code_context>

<specifics>
## Specific Ideas

- The user wants to pick a node component, choose a second node component, and
  see how connection options behave between them.
- The preview should make it easy to see the same selected node connected to
  different node types or shown inside a group.
- The canvas should support basic drag and drop so the user can move selected
  nodes and watch the connecting line and attachment slots update.
- Two selected nodes is the maximum. Lines and groups are not selected
  components.
- Computed attachment slots should handle multiple connections per side and
  label them as distributed slots, for example `top-1`, `top-2`, and `top-3`.

</specifics>
