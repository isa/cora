# Phase 3.5: Preview Visual Beauty - Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Improve the visual beauty, product feel, and first-impression polish of the
existing `cora preview` workbench.

This phase owns:
- the preview shell composition and visual hierarchy
- the center canvas surface, empty state, selection chrome, and floating tools
- the left catalog presentation and right inspector presentation
- preview-only palette, typography, borders, shadows, form-control styling, and
  motion
- screenshot/browser-verifiable visual acceptance criteria for the preview app

This phase does not own:
- canonical renderer component defaults, default component props, or component
  design language; those belong to Phase 3.6
- package surface and whether preview is distributed; that belongs to Phase 3.7
- grid capability semantics; that belongs to Phase 3.8
- YAML persistence, file watching, or interactive canvas save behavior; those
  belong to Phase 4

</domain>

<decisions>
## Implementation Decisions

### Workbench Composition
- **D-01:** The preview should use a Focused Studio direction: still a real
  workbench, but visually premium, intentional, and canvas-centered.
- **D-02:** Use the user-attached NodeFlow-style reference image as a near-match
  composition target. Broad layout should closely follow: large left library
  panel, center dotted canvas, right inspector, floating bottom tools, black
  primary action, soft white panels, restrained purple accent, and clean modern
  typography.
- **D-03:** The left panel should become a branded library panel with prominent
  Cora identity, `Library / Components` framing, segmented navigation, polished
  diagram-elements list, and a strong bottom action mapped to real preview
  behavior.
- **D-04:** The center should use a hybrid canvas: NodeFlow-style dotted studio
  background and floating composition, while preserving a clear working boundary
  where useful for testing, screenshots, and predictable canvas behavior.
- **D-05:** Keep preview as a workbench. Do not turn it into a landing page or
  marketing screen.

### Canvas Presence
- **D-06:** The canvas should emphasize a premium dotted workspace: subtle
  dotted background across the center, soft depth, floating nodes, and a quiet
  non-boxy working boundary.
- **D-07:** Selection chrome should be minimal and elegant: thin outlines, small
  neutral handles, and low visual noise. Purple remains an accent, not heavy
  editing chrome everywhere.
- **D-08:** Empty canvas should be a reference-like empty studio: mostly open
  dotted space, subtle centered hint, and floating tools present. Do not show a
  sample flow by default.
- **D-09:** Canvas controls should be split into smaller floating tool groups
  rather than one dominant pill toolbar.
- **D-10:** Remove the visible `Labels` toolbar control from primary canvas
  chrome. Attachment labels should be hidden by default or moved out of primary
  UI if still needed for debugging.

### Catalog and Inspector Polish
- **D-11:** The catalog should use a reference-style element list: larger
  polished rows, icon tiles, clear human labels, grouped under headings such as
  `Diagram Elements`, with premium spacing.
- **D-12:** Search may remain if useful, but it should not dominate the first
  impression.
- **D-13:** The inspector should use a reference-style presentation with strong
  selected-component identity: icon tile, title, small status/pill metadata,
  `Inspector / Style` visual tabs, then clean grouped controls.
- **D-14:** `Inspector / Style` tabs are visual tabs only for now. This phase
  does not require real tabbed control partitioning unless planning finds it
  trivial and low-risk.
- **D-15:** Form controls should become premium form controls: larger rounded
  inputs, cleaner labels, more whitespace, styled swatches, elegant toggles, and
  strong black/destructive action buttons where actions exist.

### Visual Language
- **D-16:** Preview UI palette should be black/white plus controlled purple
  accent, near-matching the reference. Do not use Phase 3.5 to decide renderer
  default component palette.
- **D-17:** Use soft premium panels: rounded white side panels, subtle shadows,
  light borders, and restrained depth.
- **D-18:** Typography should be large and clean: prominent Cora title,
  uppercase section labels, clear component names, and generous line-height.
- **D-19:** Remove visible generated technical labels such as `node-1`,
  `node-2`, or similar IDs from the primary UI. Technical identifiers may remain
  available only where genuinely needed for debugging or tests.
- **D-20:** Use expressive motion for premium feel: component entry, connection
  movement, toolbar/panel transitions, and selected-state feedback. Motion must
  not interfere with technical inspection.
- **D-21:** Detailed renderer design language, default components, and default
  component props are explicitly deferred to Phase 3.6.

### the agent's Discretion
- Exact copy for panel labels and buttons is open, constrained by the branded
  Cora library direction and by real preview behavior.
- Exact motion timing, easing, and implementation mechanism are open, constrained
  by usability and test stability.
- Exact responsive adaptation is open, but desktop should clearly carry the
  NodeFlow-style near-match composition.
- Exact debug affordance location for attachment labels is open, but the visible
  primary toolbar should not include the current `Labels` toggle.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/cora/src/preview/App.tsx` already separates the catalog, workbench
  canvas, and inspector areas. The near-match composition can evolve from this
  structure without replacing the app shell wholesale.
- `packages/cora/src/preview/styles.css` centralizes nearly all preview visual
  styling, so this phase can make meaningful changes with a focused CSS pass
  plus small component markup changes.
- `packages/cora/src/preview/components/WorkbenchCanvas.tsx` owns the current
  top-right toolbar, SVG border, empty `Drag components here` hint, selection
  outline, and `Labels` toggle. These are direct integration points for the
  canvas-presence decisions.
- `packages/cora/src/preview/components/CatalogSidebar.tsx` already computes a
  component list and icon glyphs. It can be restyled into a reference-style
  element list and branded library panel.
- `packages/cora/src/preview/components/NodePropPanel.tsx`,
  `ConnectionPanel.tsx`, `GroupPanel.tsx`, and `ControlInput.tsx` are the main
  inspector/form-control surfaces for premium control styling.

### Established Patterns
- Preview UI is local to `packages/cora/src/preview/`; do not add preview-only
  metadata or visual concerns into `renderer/components/`.
- The renderer components remain pure React/SVG functions. Phase 3.5 may style
  how preview frames and controls them, but should not redefine renderer
  component defaults.
- Existing preview tests focus on structure, state, geometry, and package
  assets. Visual changes should preserve behavior and update tests only where
  selectors/text intentionally change.
- The current preview uses a restrained slate/white/teal palette and simple
  bordered panels; the new direction intentionally replaces that with
  black/white/purple preview UI.

### Integration Points
- Replace the current boxy bordered SVG canvas feel with a premium dotted
  workspace while preserving deterministic SVG dimensions/viewBox behavior where
  tests rely on it.
- Move or redesign current toolbar controls into split floating tool groups.
- Remove the visible `Labels` toolbar toggle from primary UI and decide whether
  attachment labels remain hidden, debug-only, or accessible elsewhere.
- Replace visible generated node IDs in the primary inspector with human
  component names and cleaner metadata.
- Add browser/screenshot verification during execution because this is a visual
  phase.

</code_context>

<specifics>
## Specific Ideas

- The user attached a NodeFlow-style reference image and said: "I like exactly
  like this look."
- The reference qualities to preserve are: prominent app title, `Library /
  Components` framing, soft white side panels, dotted canvas field, floating
  nodes, right inspector with selected item identity, visual `Inspector / Style`
  tabs, color swatches, elegant toggles, black primary buttons, purple accent,
  and rounded premium surfaces.
- The visible preview should remove technical generated labels like node IDs
  from the main UI.
- Phase 3.5 should feel visually closer to a polished design/product tool while
  remaining a usable component preview workbench.

</specifics>
