## Research Question

What does planning need to know to implement the NodeFlow-style visual polish
for `cora preview` without breaking the existing preview workbench behavior?

## Relevant Current Code

| Area | Files | Current Shape |
|------|-------|---------------|

## Findings

### Visual Work Can Stay Preview-Local

Phase 3.5 can be implemented almost entirely inside `packages/cora/src/preview`.
The renderer component catalog should be displayed more beautifully, but the
renderer component defaults and props must remain untouched for Phase 3.6.

Implementation should avoid moving visual constants into
`renderer/components/`; preview-only tokens belong in `styles.css` or
preview-local helpers.

### Shell Changes Are Small But High-Impact

`App.tsx` already has the three major regions needed for the reference layout.
The main shell work is:

- remove or demote the current instruction strip so it does not interrupt the
  NodeFlow-style composition
- let side panels read as floating rounded white slabs, not hard gutters
- pass enough selected-item metadata to the inspector to render identity,
  visual tabs, and human component labels

### Canvas Polish Needs Both Markup and CSS

`WorkbenchCanvas.tsx` owns the visible toolbar and `Labels` toggle, so the plan
must modify it, not only CSS.

Key work:

- replace top-right toolbar with split floating control groups
- remove visible `Labels` control from primary toolbar
- keep attachment overlay label capability hidden or debug-only if retained
- make the SVG canvas visually blend into a dotted workspace while retaining
  deterministic `viewBox` and geometry behavior
- keep delete, clear, zoom, pan, drag, and selection behavior intact

### Catalog and Inspector Need Structural Hooks

The reference-style catalog needs larger row semantics and branded framing.
`CatalogSidebar.tsx` can keep `catalogItems()` and `filterComponents()` but needs
new markup/classes for:

- product title and `Library / Components` metadata
- segmented control-like navigation, visual-only if needed
- `Diagram Elements` section
- larger element rows with icon tiles
- bottom primary action only if it maps to real preview behavior

The inspector needs selected component identity. `NodePropPanel.tsx` currently
shows `definition.label` and `node.id`; `node.id` must leave the primary UI.
Connection and group inspectors need a similar identity header so the panel does
not visually collapse when the selection is not a node.

### Existing Tests Should Be Updated Rather Than Removed

`workbench.test.tsx` currently asserts:

- `Components`
- `Drag To Canvas`
- `Labels`
- `Delete`, `Clear`, `+`, `-`

These assertions will need to change because the visual contract requires:

- `Library / Components`
- `Diagram Elements`
- no visible `Labels` control
- no generated IDs in primary inspector UI
- split floating tools still exposing delete/clear/zoom through accessible
  labels or text

State and geometry tests should mostly remain unchanged. They protect the
underlying workbench behavior while the visual surface changes.

## Recommended Plan Shape

Use four plans:

1. Establish preview visual shell tokens and composition.
2. Rebuild the branded catalog and reference-style inspector surface.
3. Rework canvas surface, floating tools, minimal selection chrome, and motion.
4. Add/update visual regression-oriented verification and browser screenshot
   checks.

This keeps CSS/token work isolated first, lets catalog/inspector and canvas
proceed in parallel, then verifies the integrated desktop experience.

## Validation Architecture

Use existing Vitest preview tests plus browser/screenshot verification.

Required automated checks:

- `cd packages/cora && bun run typecheck`
- `cd packages/cora && bun x vitest run tests/preview`
- `cd packages/cora && bun run build`

Required browser verification:

- start `cora preview --no-open` or equivalent built/dev preview server
- capture a desktop screenshot
- verify: branded left library, premium dotted center, reference-style right
  inspector, no visible generated IDs, no visible `Labels` toolbar control, no
  overlapping text/controls, split floating tools

Because visual beauty is the phase goal, final verification cannot rely only on
unit tests.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Styling breaks drag/drop or selection behavior | Keep state/geometry tests green; do not rewrite state model. |
| Reference match becomes fake UI | Only add buttons/actions that map to real preview behavior; omit fake actions otherwise. |
| Phase bleeds into renderer default styling | Keep changes inside `src/preview`; do not modify renderer component defaults except through preview framing. |
| Motion makes tests flaky or inspection harder | Respect `prefers-reduced-motion`; avoid animation that changes layout or delayed geometry. |
| Screenshot verification is skipped | Make browser/screenshot verification an explicit plan and success criterion. |
