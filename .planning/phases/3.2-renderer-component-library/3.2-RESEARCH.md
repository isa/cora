# Phase 3.2: Renderer Component Library - Research

**Researched:** 2026-05-22
**Status:** Ready for planning

## Research Complete

Phase 3.2 is not a greenfield component library. It is a consistency migration
across four coupled surfaces:

1. The public renderer component barrel: `cora/renderer/components`
2. The current v1 render path: parse -> validate -> measure -> layout -> render
3. The v1 schema/examples/goldens/docs shape vocabulary
4. Future consumers: Phase 3.3 preview, Phase 4 canvas, Phase 5 icon packs

The safest plan is to build the new catalog primitives first, then migrate the
current render path to consume them, then remove old shape vocabulary from
schema/examples/goldens/docs in one deliberate compatibility-breaking cleanup.

## Existing Architecture Findings

### Current component surface

`packages/cora/src/renderer/components/index.ts` currently exports:

- Prop interfaces: `NodeComponentProps`, `EdgeComponentProps`,
  `GroupComponentProps`
- Node components: `BoxNode`, `RoundedNode`, `CylinderNode`, `CloudNode`,
  `DiamondNode`, `HexagonNode`
- Edge components: `Arrow`, `EdgeLabel`
- Group component: `Group`
- Edge decoration helpers

Phase 3.2 must replace the old public node/edge surface with:

- `Group`
- `BoxNode`
- `LabelNode`
- `IconNode`
- `LabelIconNode`
- `WebsiteNode`
- `PageNode`
- `AppNode`
- `DecisionNode`
- `IssueNode`
- `ShapeNode`
- `Line`
- reusable markers

The public barrel is already a package subpath export via
`packages/cora/package.json` and `packages/cora/tsdown.config.ts`, so no new
package export topology is needed unless files move outside
`src/renderer/components/`.

### Current render path coupling

`packages/cora/src/renderer/Diagram.tsx` dispatches on `node.shape` and imports
old components from `./components/index.js`. That means the migration has one
main render integration point:

- replace `renderNode()` shape dispatch with a catalog dispatch
- replace `<Arrow>` in the edge group with `<Line endMarker="arrow">`
- keep `<EdgeLabel>` unchanged unless line label ownership is intentionally
  folded into `Line`

`packages/cora/src/core/measureText.ts` still has shape-specific behavior:

- `shape === "diamond"` adds extra padding and forces square dimensions

If old `diamond` support is removed, `DecisionNode` needs a replacement
measurement path. The least risky approach is to introduce a catalog node kind
field before deleting old shape measurement logic. The planner should decide the
exact field name, but it should not overload `shape` with old names.

### Current schema coupling

`packages/cora/src/core/schema/diagram.schema.json` currently allows:

```text
rectangle, rounded, cylinder, cloud, diamond, hexagon, group
```

It also narrows allowed shapes by diagram kind for `box-arrows`, `flowchart`,
and `database`. Phase 3.2 decision D-03 allows schema/example/golden/docs
updates for consistency, so the planner should include a dedicated schema
migration task. Do not silently leave old shape enum values in the schema after
removing old public components.

### Current theme coupling

`packages/cora/src/renderer/themes/default.ts` uses `ThemeTokens.shapes` keys:

```text
rectangle, rounded, diamond, cylinder, cloud, hexagon, group
```

`packages/cora/src/core/themeResolver.ts` resolves node style by
`node.shape ?? "rectangle"` and expects style overrides named:

```text
fill, stroke, shadow, labelFill, strokeWidth, strokeDasharray
```

Phase 3.2 component props use the new vocabulary:

```text
backgroundColor, radius, borderStyle, borderColor, borderWidth, text,
textColor, size, iconColor, skeletonColorDark, skeletonColorLight
```

Research recommendation: do not try to fully replace `ThemeTokens` in the same
step as catalog creation. Instead:

1. Define normalized component prop/types and pure component helpers.
2. Add adapter helpers that derive those props from current layout/theme data.
3. Migrate schema/examples/render path after the adapters exist.
4. Leave full provider theme semantics to Phase 5.

This keeps current PDF overlay and layout behavior stable while still making the
public component API consistent.

## SVG and React Findings

### Markers

SVG markers are the correct primitive for reusable line endpoint markers.
External SVG references confirm that `<marker>` is designed for marker graphics
attached to paths/lines/polylines/polygons via `marker-start`, `marker-mid`, and
`marker-end`. The implementation should centralize marker defs in one module and
avoid duplicating arrow/circle geometry in each line component.

Recommended shape:

- `MarkerType = "none" | "arrow" | "circle" | "filledCircle"`
- `LineMarkerProps = { startMarker?: MarkerType; endMarker?: MarkerType }`
- `LineMarkers` or `MarkerDefs` component emits stable ids in `<defs>`
- `Line` references marker ids with `markerStart` / `markerEnd`

Implementation caution: marker ids are global within an SVG document. The ids
must be stable and collision-safe for the renderer. Since Cora owns the whole SVG
tree today, deterministic ids like `cora-marker-arrow` are acceptable. If Phase
3.3 renders multiple isolated previews in the same DOM, preview code may need
id prefixing; that should not leak into the raw renderer components unless the
planner chooses an explicit `idPrefix` prop.

### Line styles

SVG stroke dash patterns are the correct implementation primitive for
`lineStyle: "solid" | "dashed" | "dotted"`.

Recommended mapping:

- `solid` -> no `strokeDasharray`
- `dashed` -> `strokeDasharray="6 4"` or a similar constant scaled to
  `strokeWidth`
- `dotted` -> `strokeDasharray="1 4"` plus round line caps if the output should
  look like dots rather than square dashes

Keep the mapping centralized so both `Line` and any future border helpers can
reuse the same style conversion logic.

### Static rendering

Cora already uses React server rendering through `renderToStaticMarkup`.
React's current API renders a non-interactive static HTML string, which fits
Cora's headless SVG export model. Phase 3.2 components must remain pure
React/SVG functions and must not depend on browser APIs, hydration, or effects.

## Type Contract Recommendations

Add explicit component-library props in `components/types.ts` or split by
family:

```text
type BorderStyle = "none" | "solid" | "dashed" | "dotted";
type SizePreset = "sm" | "md" | "lg" | "xl" | "xxl";
type ComponentSize = SizePreset | { width: number; height: number };
type MarkerType = "none" | "arrow" | "circle" | "filledCircle";
type PageNodeType = "landing" | "form" | "content" | "profile" | "settings";
type IssueIconType = "bug" | "warning" | "error" | "stop";
```

`BoxStyleProps`:

```text
backgroundColor?: string;
radius?: number;
borderStyle?: BorderStyle;
borderColor?: string;
borderWidth?: number;
text?: string;
textColor?: string;
size?: ComponentSize;
```

Do not remove the existing hybrid props until the current render path has a
clean adapter:

```text
NodeComponentProps
EdgeComponentProps
GroupComponentProps
```

The new catalog props and the current layout-IR props can coexist if the public
barrel clearly exports both as separate contracts.

## Icon Slot Contract

Phase 3.2 should define the component-side icon contract without implementing
extension loading. Recommended type:

```text
type SvgIconComponent = (props: {
  x?: number;
  y?: number;
  size: number;
  color: string;
  title?: string;
}) => React.ReactNode;
```

or a React component equivalent. `IconNode` receives an icon renderer/function
directly. Phase 5 can later resolve provider/service metadata into that icon
renderer.

Avoid requiring icon ids or extension ids in raw components. That would couple
components to Phase 5's registry design too early.

## Migration Strategy

### Step 1: Add catalog primitives without changing current render behavior

Create `BoxStyleProps`, size helpers, border helpers, icon contract, `Line`,
marker defs, and all named catalog nodes. Add unit tests for helpers/components
using `renderToStaticMarkup` string assertions.

This step should not touch schema/examples/goldens yet.

### Step 2: Migrate current render path to catalog adapters

Update `Diagram.tsx` so:

- current rectangular/rounded paths render through `BoxNode`
- decision-like nodes render through `DecisionNode`
- existing edge paths render through `Line` with `endMarker="arrow"`
- `Arrow` is removed from the public barrel or marked internal compatibility

If old schema values still exist during this step, keep mapping local and
temporary. Do not present old shape components as public catalog members.

### Step 3: Remove old shape vocabulary consistently

Update:

- `diagram.schema.json`
- `core/types.ts` if a new field is needed
- `measureText.ts`
- `themeResolver.ts`
- `default.ts`
- examples in `examples/valid/`
- golden SVGs
- `AGENTS.md`
- public barrel exports

The exact YAML shape is a planning decision, but the implementation must avoid
a half-state where docs say old shapes are gone while schema still accepts them.

## Risks and Mitigations

### Risk: schema migration breaks current examples

Mitigation: make schema/examples/goldens one atomic task. After the task,
`bun run cora validate examples/valid/*.yaml --format json` should pass for
every valid fixture.

### Risk: removing `diamond` breaks flowchart measurement

Mitigation: introduce `DecisionNode` measurement behavior before removing the
old `shape === "diamond"` branch. Decide whether measurement keys are based on
a new `node.type`/`node.component` field or a normalized internal mapper.

### Risk: marker ids collide in preview

Mitigation: centralize marker ids and allow future id prefixing at the marker
defs layer. Do not scatter `<marker>` definitions inside individual lines.

### Risk: icon contract accidentally implements Phase 5

Mitigation: accept an icon renderer/slot directly in `IconNode` and
`LabelIconNode`; do not implement extension registry lookup, provider/service
resolution, installation, caching, or lockfiles.

### Risk: public API churn

Mitigation: keep old hybrid prop interfaces available if the current render
path still needs them, but remove old public shape component exports and make
the catalog prop types explicit.

## Suggested Plan Breakdown

### Plan 01: Catalog Types, Style Helpers, Line, and Markers

Build reusable types/helpers and the new `Line` primitive with marker defs and
tests. No schema/example migration yet.

Requirements: RCL-03, RCL-04, RCL-08, RCL-09, RCL-10.

### Plan 02: Node Catalog Components and Icon Slot Contract

Implement named node catalog components, `ShapeNode`, shared helpers, and
component-level tests.

Requirements: RCL-01, RCL-03, RCL-04, RCL-05, RCL-06, RCL-07, RCL-10.

### Plan 03: Current Renderer Migration and Public Barrel Cleanup

Update `Diagram.tsx` and the public barrel to use/export the new catalog and
make `Arrow` internal compatibility only.

Requirements: RCL-01, RCL-02, RCL-08, RCL-09, RCL-10.

### Plan 04: Schema, Examples, Goldens, and Documentation Consistency

Remove old shape vocabulary from schema/examples/docs, refresh goldens, and
update AGENTS.md. This should be the final consistency sweep.

Requirements: RCL-02 plus documentation/verification coverage for all RCL IDs.

## Validation Architecture

### Dimensions

1. **Public API coverage:** all catalog components and prop types are exported
   from `cora/renderer/components`.
2. **Old shape removal:** old public shape-specific components and old shape
   values are absent from public barrel, schema, examples, and AGENTS.md.
3. **Pure SVG compatibility:** components render through `renderToStaticMarkup`
   and do not use DOM APIs/effects.
4. **Line/marker correctness:** `Line` renders routed points with correct dash
   and marker attributes.
5. **Current render path compatibility:** `cora render` still validates and
   renders every valid example.
6. **Export artifact integrity:** `dist/renderer/components/index.d.ts` is
   emitted by build.
7. **Regression safety:** golden SVGs and PDF tests remain green after the
   intentional model migration.
