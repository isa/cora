# Phase 3.2: Renderer Component Library - Context

**Gathered:** 2026-05-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the full reusable renderer component catalog and normalized style
vocabulary under `packages/cora/src/renderer/components/` so the preview,
interactive canvas, and extension phases grow around a consistent public
component model.

This phase intentionally replaces the old shape-specific renderer model with
the new catalog. It may update schema/examples/goldens/docs where needed to
keep the public model consistent. It must not build the Phase 3.3 preview UI or
the Phase 5 extension loader.

</domain>

<decisions>
## Implementation Decisions

### Catalog shape
- **D-01:** Fold the existing shape model into the new component catalog rather
  than keeping old and new public models side by side.
- **D-02:** Remove old public shape-specific components and old shape values for
  consistency. `CylinderNode`, `CloudNode`, `DiamondNode`, and `HexagonNode`
  should not remain as public catalog components.
- **D-03:** Phase 3.2 may update schema, examples, goldens, and docs together
  if required to remove old shape support cleanly. Consistency across the
  product is more important than preserving legacy shape names.
- **D-04:** The canonical catalog includes `Group`, `BoxNode`, `LabelNode`,
  `IconNode`, `LabelIconNode`, `WebsiteNode`, `PageNode`, `AppNode`,
  `DecisionNode`, `IssueNode`, `ShapeNode`, `Line`, and reusable markers.
- **D-05:** `ShapeNode` is a low-level escape hatch for future odd shapes, but it
  must not weaken the consistency of the named catalog.

### Style prop contract
- **D-06:** All box-like components share one base `BoxStyleProps` type:
  `backgroundColor`, `radius`, `borderStyle`, `borderColor`, `borderWidth`,
  `text`, `textColor`, and `size`.
- **D-07:** `borderStyle` values are `none | solid | dashed | dotted`.
- **D-08:** `size` supports both explicit dimensions and presets:
  `{ width: number; height: number } | "sm" | "md" | "lg" | "xl" | "xxl"`.
- **D-09:** Components add only their specific extras on top of shared box-like
  props, such as `iconColor`, `type`, `skeletonColorDark`, or
  `skeletonColorLight`.
- **D-10:** Use TypeScript-friendly camelCase names in code, not YAML-style
  hyphenated names.

### Specialized components
- **D-11:** `PageNode.type` values are `landing | form | content | profile |
  settings`.
- **D-12:** `IssueNode.icon` values are `bug | warning | error | stop`.
- **D-13:** `PageNode` uses `skeletonColorDark` and `skeletonColorLight`.
- **D-14:** `WebsiteNode` and `AppNode` are box-like and support optional
  `text`.
- **D-15:** Box-driven nodes may bear icons where appropriate.
- **D-16:** `IconNode` is icon-only. It renders an SVG icon from the future icon
  library/pack contract, not a text/box hybrid.
- **D-17:** Phase 3.2 should define the component-side icon slot/contract. The
  actual extension pack loading and install flow remains Phase 5.

### Lines and markers
- **D-18:** `Line` receives explicit routed SVG points:
  `points: Array<{ x: number; y: number }>`.
- **D-19:** Routing stays outside the component. `Line` renders the given path,
  style, and markers only.
- **D-20:** Marker values are canonical TypeScript literals:
  `none | arrow | circle | filledCircle`.
- **D-21:** `Line` replaces `Arrow` publicly. `Arrow` may remain internal only
  as a compatibility adapter while current edge rendering is migrated.
- **D-22:** Existing arrow-style edges can map to `Line` with
  `endMarker: "arrow"` during the transition.

### the agent's Discretion
- Exact file split inside `components/` is open, as long as the public barrel
  remains `renderer/components/index.ts` and the code is organized by family.
- Exact default values for preset sizes, colors, radii, marker dimensions, and
  icon fallback rendering are planner/executor decisions, constrained by pure
  SVG output and golden tests.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/cora/src/renderer/components/` already exists as the real
  implementation home and package subpath surface.
- Existing components are pure React/SVG functions and can be used as migration
  references.
- `NodeLabel`, `ShapeShadow`, and `strokeWidth` in
  `components/nodes/shared.tsx` are useful shared rendering primitives.
- Current edge geometry utilities in `packages/cora/src/core/edgeGeometry.ts`
  already produce routed points suitable for a `Line` component.

### Established Patterns
- Package consumers use `cora/renderer/components`; top-level `cora` remains
  free of component exports.
- `Diagram.tsx` consumes only `./components/index.js`; preserve that boundary.
- Golden SVG tests are the render-output regression contract.
- Pure SVG and DOM-free measurement are non-negotiable.

### Integration Points
- `components/index.ts` expands to export the new catalog, style types, line
  types, marker types, and icon-slot types.
- `Diagram.tsx` must keep working with the current v1 render path while it maps
  old edge rendering to the new `Line` primitive.
- `package.json` and `tsdown.config.ts` already build and export
  `dist/renderer/components/index.{js,d.ts}`.
- Schema/examples/goldens/docs should be changed in the same plan if old shape
  values are removed.

</code_context>

<specifics>
## Specific Ideas

- The user wants consistency now, before the codebase gets larger.
- `IconNode` should be icon-only and should use SVG icons from a future library
  or pack system.
- Box-driven nodes may optionally carry icons, but they are distinct from the
  icon-only node.
- `WebsiteNode` and `AppNode` should not be forced to be textless; optional
  labels are allowed.

</specifics>

<deferred>
## Deferred Ideas

- Phase 3.3 owns the preview UI, live controls, component pack browser, and
  line-attachment overlay.
- Phase 5 owns extension installation, icon pack loading, provider/service icon
  resolution, and `cora-extensions` integration.
- YAML schema mapping for future extension packs should stay outside the raw
  component layer, even if Phase 3.2 updates current shape/schema vocabulary for
  consistency.

</deferred>
