# Phase 3.8: Grid Capability Expansion - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Strengthen Cora's diagram grid behavior before Phase 4 interactive canvas: define a
canonical coordinate grid, add `diagram.grid` to the YAML schema, implement shared
snap math in core, wire snap UX into the preview workbench, and document semantics
so `cora serve` can reuse the same foundation without re-deciding grid rules.

This phase owns:
- grid cell size, origin, and visual hierarchy defaults
- schema fields for diagram-level grid configuration
- core snap/round helpers consumed by preview (and later canvas)
- preview toolbar toggle + Shift override + live snap on nodes/groups
- tests and docs locking grid behavior for Phase 4

This phase does not own:
- YAML AST save/writeback of snapped positions (Phase 4 — semantics locked here)
- alignment guides to other nodes (deferred)
- visible grid in SVG/PDF/PNG exports
- ELK post-layout grid snapping by default
- text/ASCII export grid behavior (must remain unaffected)

</domain>

<decisions>
## Implementation Decisions

### Grid coordinate model
- **D-01:** Canonical grid **cell size is 16px**. Update preview canvas visual
  grid from the current decorative 24px CSS dots to match 16px spacing (or
  derive preview dots from the same constant as core snap math).
- **D-02:** Grid **origin is fixed at world (0, 0)**. Snap math uses absolute
  diagram coordinates — not viewport-relative or content-bbox-relative origins.
  (Claude discretion — best fit for YAML `position` persistence and Phase 4 save.)
- **D-03:** Add **`diagram.grid` to the JSON Schema now** as optional diagram-level
  configuration (spacing, visibility toggles for edit surfaces, etc.). Agents and
  humans can express grid prefs in YAML; defaults apply when omitted.
- **D-04:** Edit-surface visual grid uses **major lines every 5 cells** (80px at
  16px spacing) for hierarchy; minor lines at cell boundaries. Static exports do
  not render these lines.

### Snap behavior policy
- **D-05:** Snapping is controlled by a **toolbar toggle, default ON**. Holding
  **Shift temporarily disables snap** (inverse override while toggle is on).
- **D-06:** **Nodes and groups snap** on move and resize. **Connection endpoint
  drags remain free-form** in Phase 3.8.
- **D-07:** **Grid-only snapping** — no alignment guides to other node
  edges/centers in this phase.
- **D-08:** Snap applies **live during drag** — positions update continuously to
  snapped coords while dragging, not only on pointer release.

### Static render path
- **D-09:** Grid is **edit-time only**. SVG, PDF, and PNG exports **never** show
  a grid background regardless of `diagram.grid` settings.
- **D-10:** **ELK auto-layout does not snap** its output to the grid by default.
  Layout engine positions stay as computed to preserve layout quality and golden
  stability. (Claude discretion — optional future `snapLayout` field may default
  false; not required in Phase 3.8 execution unless planner finds it trivial.)
- **D-11:** Existing **`layout: preserve | hybrid` YAML positions stay exact** in
  the render pipeline until Phase 4 canvas **save rounds positions to grid** on
  writeback. Phase 3.8 documents this contract; save implementation is Phase 4.
- **D-12:** **Text/ASCII export is fully unaffected** by grid — no grid metadata,
  snapping, or visual affordances leak into `renderToText.ts`.

### Preview vs shared core delivery
- **D-13:** **Core module first** — implement snap/round helpers before preview
  wiring so Phase 4 reuses the same module unchanged.
- **D-14:** Grid math lives in **`packages/cora/src/core/grid.ts`** alongside
  layout coordinate conventions.
- **D-15:** Grid utilities are **public via the `cora/core` subpath export** for
  extension authors and future canvas code — not top-level `cora` barrel exports.
- **D-16:** Phase 3.8 ships **full stack**: schema (`diagram.grid`), core helpers,
  preview snap UI (toggle + Shift + live snap), tests/fixtures, and Phase 4-facing
  docs describing grid semantics.

### Claude's Discretion
- Exact `diagram.grid` schema property names and defaults (e.g., `spacing: 16`,
  `majorEvery: 5`, edit-surface visibility flags).
- Whether preview grid dots use CSS only or SVG overlay aligned to world coords.
- Exact toolbar control placement and keyboard shortcut labeling in preview.
- Unit test vs integration test split for snap math and preview drag behavior.
- Whether to add a non-default `snapLayout` opt-in for ELK (default false) or
  defer entirely.

</decisions>

<specifics>
## Specific Ideas

- User chose **16px over 24px** despite preview CSS currently using 24px dots — visual grid must be updated to match snap math.
- Snap UX combines **toggle default ON** with **Shift to temporarily disable** — both behaviors required, not either/or.
- **Major lines every 5 cells** (80px) for edit-surface visual hierarchy.
- Phase 4 save should **round positions to grid on writeback**; until then preserve/hybrid YAML coords render exactly as authored.

</specifics>

<deferred>
## Deferred Ideas

- **Alignment guides** — snap to other node edges/centers; belongs in Phase 4 interactive canvas polish.
- **Connection endpoint grid snap** — free-form in Phase 3.8; revisit if canvas editing needs it.
- **Optional ELK post-layout snap** (`snapLayout: true`) — not default; defer unless trivial during implementation.
- **Visible grid in static exports** — explicitly rejected (edit-time only).

</deferred>
