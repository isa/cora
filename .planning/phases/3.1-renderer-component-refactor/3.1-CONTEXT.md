# Phase 3.1: Renderer Component Refactor - Context

**Gathered:** 2026-05-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract the current renderer's React node, edge, and group components into a stable `packages/cora/src/renderer/components/` component surface without changing existing YAML render behavior.

This phase is a structural refactor of the current renderer. It must prepare a public component contract for future preview/canvas/extension work, but it must not implement the broader component catalog. The full catalog is now Phase 3.2: Renderer Component Library.

</domain>

<decisions>
## Implementation Decisions

### Public component API
- **D-01:** Use a hybrid public API: stable interface names (`NodeComponentProps`, `EdgeComponentProps`, `GroupComponentProps`) while carrying the current `LayoutedNode`, `LayoutedEdge`, `LayoutedGroup`, and `ThemeTokens` objects inside those props.
- **D-02:** Treat those prop interfaces as the public extension target now, not merely private renderer internals.
- **D-03:** Export the public surface from `packages/cora/src/renderer/components/index.ts`; `Diagram.tsx` must consume components only through this barrel.
- **D-04:** Expose package consumers through a dedicated subpath export (`cora/renderer/components`) if package `exports` blocks deep imports. Do not re-export these interfaces from `packages/cora/src/index.ts` in this phase.
- **D-05:** Document the component API as a supported v1 renderer contract with room for minor documented evolution before 1.0; do not claim it is frozen forever.

### Component location
- **D-06:** Component implementations live under `packages/cora/src/renderer/components/`, not in the old `renderer/nodes`, `renderer/edges`, and `renderer/groups` folders.
- **D-07:** Organize the implementation by family:
  - `renderer/components/nodes/`
  - `renderer/components/edges/`
  - `renderer/components/groups/`
  - `renderer/components/types.ts`
  - `renderer/components/index.ts`
- **D-08:** Existing old folders may be removed or reduced to compatibility shims only if needed for internal imports; after this phase, `Diagram.tsx` must not deep-import from old folders.

### Future component catalog
- **D-09:** The broader component vocabulary is intentionally deferred to Phase 3.2, but Phase 3.1 must not block it. The future catalog includes groups, box/label/icon/page/app/decision/issue nodes, configurable lines, and endpoint markers.
- **D-10:** Normalize future style vocabulary toward TypeScript-friendly names: `backgroundColor`, `radius`, `borderStyle`, `borderColor`, `borderWidth`, `text`, `textColor`, `size`, `lineStyle`, `strokeColor`, `strokeWidth`, `startMarker`, `endMarker`.
- **D-11:** `label node` uses `text` (not `ext`).

### Baseline repair policy
- **D-12:** Phase 3.1 owns the inherited renderer baseline issues from Phase 3: current golden SVG drift and `measureText.ts` typecheck errors.
- **D-13:** The executor should refresh goldens before moving components, because the drift is pre-existing. The refreshed baselines become the contract for the structural refactor that follows.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Current renderer components are already stateless React/SVG functions.
- `Diagram.tsx` composes groups, nodes, edges, and edge labels; this is the only composition root that should need import rewiring.
- `layout-ir.ts` already provides `LayoutedNode`, `LayoutedEdge`, `LayoutedGroup`, and `ThemeTokens` types that can back the hybrid prop interfaces.
- `tests/render-golden.mjs` can refresh and verify all five current diagram kinds.

### Established Patterns
- Package exports currently include `.` / `./core` / `./renderer`; deep imports may be blocked for downstream consumers unless `./renderer/components` is added.
- Existing plan files use focused execution plans with explicit `read_first`, `action`, `acceptance_criteria`, and `verify` blocks.
- Phase 3 used `bun x vitest run`, `bun run build`, and smoke scripts as verification gates.

### Integration Points
- `Diagram.tsx` should import `Arrow`, `EdgeLabel`, `Group`, node components, and prop types from `./components/index.js`.
- `renderToSVG.tsx` should continue exporting `Diagram` as before.
- `renderer/index.ts` can continue exporting renderer functions; component exports should stay on `renderer/components/index.ts` and package subpath `cora/renderer/components`.

</code_context>

<specifics>
## Specific Ideas

- Make `renderer/components/` the real implementation home, not only a re-export facade.
- Keep this phase output-stable except for the explicit golden baseline refresh needed to repair inherited drift.
- Leave the full component library catalog to Phase 3.2, where it can be designed without hiding inside a refactor.

</specifics>
